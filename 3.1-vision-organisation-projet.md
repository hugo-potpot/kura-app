# Document 3.1 — Vision idéale de l'organisation du projet KURA

**Auteur :** Hugo Potier  
**Projet :** KURA (idel-app) — Application mobile pour infirmiers libéraux  
**Date :** Mai 2026

---

## 1. Compréhension du projet

### 1.1 Contexte et problème résolu

KURA est une application mobile **offline-first** destinée aux 120 000+ infirmiers libéraux (IDEL) en France. Le constat de départ est fort : les IDEL perdent entre 30 et 60 minutes par jour en organisation manuelle de leurs tournées, avec des outils fragmentés et vieillissants (Vega, Ozzen, agendas papier) qui ne répondent pas aux réalités terrain — en particulier dans les zones à faible couverture réseau où 66 % des professionnels rencontrent des difficultés régulières.

La connexion personnelle au métier (ma mère et ma sœur sont IDEL) et le sondage mené auprès de 12 professionnels ont permis d'ancrer le projet dans des besoins réels et mesurables.

### 1.2 La solution proposée

KURA repose sur quatre innovations combinées qu'aucun concurrent ne propose à ce jour :

1. **Planning IA contextuel** — algorithme d'optimisation de tournée (TSP) adapté aux contraintes métier IDEL : durée variable des soins, créneaux horaires patients, préférences géographiques.
2. **Transmissions vocales IA** — transcription on-device via Whisper (0 cloud, conforme HDS), avec validation humaine obligatoire.
3. **Mode offline natif** — 100 % fonctionnel sans réseau (SQLite local chiffré AES-256 / PostgreSQL serveur HDS).
4. **IA assistante** — l'algorithme suggère, l'IDEL décide toujours. Conformité déontologique garantie.

### 1.3 Périmètre technique et réglementaire

Le projet est **greenfield**, classifié *mobile healthcare app — complexité haute*. Il couvre 84 exigences fonctionnelles réparties sur 8 domaines : authentification (MFA/FIDO2), gestion des structures multi-tenant, gestion des patients (RGPD), planning intelligent, transmissions, mode offline/sync, notifications push, et back office web (Next.js).

La contrainte réglementaire est structurante : hébergement HDS certifié obligatoire, chiffrement AES-256 au repos, audit trail immuable 3 ans, isolation multi-tenant stricte. Ces exigences ne sont pas négociables et impactent l'ensemble des choix d'architecture.

## 2. Composition idéale de l'équipe

### 2.1 Analyse des compétences nécessaires

Un projet de cette nature — mobile healthcare, offline-first, IA embarquée, compliance HDS/RGPD — requiert idéalement de couvrir les domaines suivants :

| Domaine | Justification |
|---|---|
| Développement mobile (React Native / Expo) | Cœur de l'application, 80 % de l'usage |
| Développement web / API (Next.js) | Back office admin + API REST |
| Architecture backend / base de données | PostgreSQL, sync bidirectionnel, isolation multi-tenant |
| Sécurité / conformité HDS-RGPD | Non-négociable dans le domaine médical |
| UX Design | Utilisateurs terrain non techniques, interface doit être intuitive |
| DevOps / infra | Déploiement EAS, hébergeur HDS, CI/CD |
| Gestion de projet | Coordination, livrables, suivi client |

### 2.2 Organisation pour une équipe de 2

Dans le cadre de ce PFE à deux, j'aurais organisé les rôles de la manière suivante :

**Hugo (développeur 1 — rôle principal) :**
- **Lead mobile** : architecture React Native / Expo, navigation (Expo Router v4), synchronisation offline, algorithme de planning TSP
- **Scrum Master** : facilitation des cérémonies agiles, gestion du backlog, suivi des livraisons
- **Product Owner proxy** : priorisation des stories, relation avec les utilisateurs terrain (ma famille IDEL), arbitrages fonctionnels
- **Responsable sécurité** : conformité HDS, gestion des secrets, audit trail

**Co-développeur (développeur 2 — rôle secondaire) :**
- **Lead web / API** : Next.js back office, routes API REST, authentification BetterAuth, RBAC
- **Lead design** : maquettes Figma, cohérence du design system (React Native Paper), expérience utilisateur
- **DevOps** : pipeline CI/CD, déploiement EAS Build, gestion base de données PostgreSQL HDS

**Partage transversal :**
- Les deux développeurs partagent la propriété du schéma Drizzle (`packages/db/schema/`) et des packages partagés (`packages/shared`)
- Les décisions d'architecture sont collégiales, documentées dans le fichier `architecture.md`
- La revue de code (PR) est croisée : chaque merge nécessite une validation par le second développeur

### 2.3 Instances de pilotage

Même avec 2 personnes, j'aurais instauré deux instances distinctes :

- **Stand-up quotidien (10 min)** : synchronisation rapide sur les blocages et l'avancement du jour
- **Sprint review / retrospective (bi-hebdomadaire, 1h)** : démo des fonctionnalités livrées, ajustement du backlog, bilan méthodologique
- **Point client mensuel (30 min)** : retour terrain des IDEL testeurs, validation des choix UX, ajustement des priorités

## 3. Méthode de management choisie

### 3.1 Justification du choix : Scrum allégé

J'aurais opté pour un **Scrum allégé à 2 personnes**, plutôt qu'un Kanban pur ou une méthode en cascade, pour les raisons suivantes :

- Le projet est **greenfield** avec des exigences évolutives — la cascade est inadaptée (on ne peut pas tout spécifier en amont dans le domaine médical).
- Le **Kanban pur** manque de structure temporelle : sans sprints, il est difficile d'évaluer la vélocité et de tenir des engagements de livraison vis-à-vis du jury.
- Le **Scrum** impose des rituels (sprint planning, review, retro) qui forcent à prioriser, à livrer régulièrement, et à s'adapter. C'est précisément ce qu'exige un projet étudiant avec une date de soutenance fixe.

L'allègement concerne les cérémonies : pas de story poker formel (estimation directe en points par les 2 développeurs), pas de daily physique (message quotidien sur Discord suffit à 2), et product backlog tenu directement dans les fichiers YAML du projet.

### 3.2 Structure des sprints

- **Durée de sprint** : 2 semaines
- **Vélocité cible** : 8 à 12 story points par sprint (ajustée après les 2 premiers sprints)
- **Definition of Done** : code reviewé + testé (unitaire et E2E si applicable) + déployé sur environnement de staging + documentation inline à jour

### 3.3 Outil de suivi

J'aurais centralisé le suivi dans **GitHub Projects** (lié au dépôt), avec :
- Un tableau Kanban par sprint : `Todo / In Progress / In Review / Done`
- Des issues liées directement aux user stories documentées (ex : `Story 4.2 - Génération automatique du planning`)
- Les milestones GitHub correspondant aux fins de sprint et aux jalons soutenance

### 3.4 Évaluation continue

Chaque sprint se termine par une **métrique de livraison** (pourcentage de stories Done vs planifiées) et une **métrique qualité** (nombre de bugs régressifs découverts lors de la review). Si le ratio stories livrées / planifiées descend sous 70 % deux sprints de suite, le backlog est repriorisé immédiatement.

## 4. Organisation idéale du planning

### 4.1 Découpage en phases

| Phase | Période | Contenu | Jalon |
|---|---|---|---|
| **Phase 0 — Setup** | Semaines 1–2 | Monorepo Turborepo, CI/CD, DB schema, conventions | Socle technique opérationnel |
| **Phase 1 — Auth & Structure** | Semaines 3–6 | Authentification MFA/FIDO2, gestion des structures, invitations | Login complet fonctionnel |
| **Phase 2 — Patients** | Semaines 7–10 | CRUD patients, géolocalisation, multi-tenant, RGPD | Gestion patients offline |
| **Phase 3 — Planning IA** | Semaines 11–14 | Algorithme TSP, drag & drop, urgences, carte | Planning généré < 5s |
| **Phase 4 — Transmissions** | Semaines 15–18 | Saisie vocale Whisper, templates, audit trail, constantes | Transmissions offline |
| **Phase 5 — Offline & Sync** | Semaines 19–22 | Queue Zustand persistée, sync bidirectionnel, résolution conflits | 100 % fonctionnel hors ligne |
| **Phase 6 — Back Office** | Semaines 23–24 | Interface web admin, import CSV, dashboard | Back office livrable |
| **Phase 7 — Polish & Soutenance** | Semaines 25–26 | Tests terrain, corrections, préparation démo | Soutenance |

### 4.2 Critères de priorisation (MoSCoW)

Les 84 exigences fonctionnelles ont été découpées selon le principe MoSCoW :

- **Must have** (bloquant soutenance) : auth MFA, patients offline, planning IA, transmissions vocales, sync offline
- **Should have** (valeur démontrée) : back office web, notifications push, graphiques constantes
- **Could have** (bonus si temps) : import CSV, export RGPD automatisé, mode manuel pur
- **Won't have** (post-MVP) : facturation CPAM, scan ordonnances OCR, analytics avancés

### 4.3 Gestion de la dette technique

J'aurais réservé **10 % de la capacité de chaque sprint** à la dette technique : refactoring, amélioration de la couverture de tests, mise à jour des dépendances. Cette règle évite l'accumulation silencieuse qui ralentit les sprints finaux.

## 5. Vision de la gestion des échanges

### 5.1 Outils de communication internes

| Canal | Usage | Fréquence |
|---|---|---|
| **Discord** (channel #daily) | Stand-up quotidien asynchrone, blocages | Quotidien |
| **Discord** (channel #tech) | Questions techniques, partage de ressources | À la demande |
| **GitHub Pull Requests** | Revue de code, discussion sur les choix d'implémentation | À chaque PR |
| **GitHub Issues** | Suivi des bugs, des stories, des décisions | Continu |
| **Fichiers `architecture.md` / `project-context.md`** | Documentation des décisions d'architecture | Mis à jour à chaque décision structurante |

### 5.2 Partage de l'information

Toute décision technique structurante (choix d'une librairie, d'un pattern d'architecture, d'une stratégie de sync) est immédiatement documentée dans le fichier `project-context.md`. Cela garantit qu'aucune connaissance ne reste "dans la tête" d'un seul développeur — essentiel dans une équipe de 2 où l'absence de l'un peut bloquer l'autre.

### 5.3 Organisation des flux d'information avec les parties prenantes extérieures

- **IDEL testeurs terrain** : invitées en fin de phase (review sprint) pour valider les parcours utilisateurs critiques (planning, transmission d'une constante). Feedback collecté via formulaire structuré, pas d'interview libre à chaque sprint.
- **Jury / encadrants** : livrables documentés (PRD, architecture, epics/stories, sprint reports) accessibles sur le dépôt Git. Point de synchronisation mensuel ou à chaque jalon majeur.
- **Aucun outil externe lourd** (Jira, Confluence, Notion) : un projet étudiant à 2 personnes doit optimiser le temps de développement, pas l'outillage de gestion.

## 6. Gestion du client

### 6.1 Qui est le client ?

Dans ce projet, le "client" est double :

1. **Le jury / les encadrants académiques** — destinataires des livrables formels (documents, soutenance, démonstration).
2. **Les IDEL utilisatrices finales** — les vraies bénéficiaires du produit, représentées par les personas Marie (80 %), Sophie (15 %) et Michel (5 %) définis lors de la phase de discovery.

### 6.2 Implication du client tout au long du projet

J'aurais structuré l'implication des utilisateurs en trois moments clés :

**Avant (discovery) :**  
Sondage structuré auprès de 12 IDEL (réalisé en amont), entretiens qualitatifs semi-directifs (30 min) avec 3 profils représentatifs pour valider les personas et hiérarchiser les irritants. Ces données alimentent directement le PRD et les critères de succès.

**Pendant (itérations) :**  
Test utilisateur sur prototype (maquettes Figma) à l'issue de la phase UX, avant tout développement. Puis test fonctionnel sur build mobile (via TestFlight / EAS) en fin de phase 3 (planning) et phase 4 (transmissions) — les deux features à plus haute valeur perçue. 4 IDEL testeurs minimum, session de 45 minutes avec scenario guidé.

**Après (validation finale) :**  
Session de validation terrain complète avant la soutenance : 4 testeurs, scenarii couvrant le golden path complet (login → planning généré → visite patient → transmission → sync). L'objectif explicite du MVP est que 4/4 testeurs valident un gain de temps réel perçu.

### 6.3 Prise de décision collégiale

Toute modification fonctionnelle sortant du périmètre MVP — ajout de feature, changement de priorité, décision d'architecture impactant la roadmap — est arbitrée conjointement par les deux développeurs, documentée dans les issues GitHub, et communiquée au jury si elle affecte les livrables académiques.

Le client (jury) est informé en amont des ajustements de périmètre, pas après coup. Un delta entre le PRD initial et l'implémentation finale est acceptable à condition d'être explicité et justifié.

## 7. Anticipation des risques

### 7.1 Registre des risques identifiés

| # | Risque | Probabilité | Impact | Stratégie |
|---|---|---|---|---|
| R1 | Complexité du sync offline bidirectionnel | Haute | Critique | Prototypage dès la phase 0 ; stratégie "serveur gagne" pour simplifier la résolution de conflits |
| R2 | Conformité HDS non satisfaite à la soutenance | Moyenne | Critique | Architecture contrainte dès le début (pas de refactoring HDS en fin de projet) ; choix d'un hébergeur HDS certifié dès le sprint 1 |
| R3 | Algorithme TSP lent (> 5s pour 15 patients) | Moyenne | Élevé | Exécution locale sur appareil (pas de round-trip réseau) ; optimisation greedy + nearest-neighbor en fallback ; tests de performance automatisés |
| R4 | API de géocodage indisponible ou lente | Moyenne | Moyen | Géocodage asynchrone en arrière-plan, fallback badge "adresse non localisée", Nominatim (gratuit, RGPD) en priorité |
| R5 | Intégration Whisper on-device non stable | Haute | Élevé | Feature flag : si Whisper échoue, basculement transparent sur saisie textuelle sans bloquer l'utilisateur |
| R6 | Dérapage du scope (feature creep) | Haute | Élevé | Backlog verrouillé dès le sprint 4 ; toute demande post-lock tombe en post-MVP explicitement |
| R7 | Défaillance d'un développeur (maladie, abandons) | Faible | Critique | Bus factor = 0 : connaissance partagée via `project-context.md`, revue de code systématique, pas de domaine "propriété" d'une seule personne |
| R8 | Données patients exposées (fuite, mauvaise isolation multi-tenant) | Faible | Critique | `structure_id` obligatoire sur toutes les requêtes Drizzle ; tests d'isolation multi-tenant automatisés ; pas de données patients dans les logs ni les notifications push |
| R9 | Incompatibilité EAS Build / Expo SDK en production | Moyenne | Élevé | Gel des versions majeures (Expo SDK 53, EAS CLI fixée) dès le sprint 0 ; mise à jour uniquement entre phases, jamais en milieu de sprint |
| R10 | Mauvaise adoption par les testeurs terrain | Faible | Moyen | UX validée sur maquettes avant développement ; utilisateurs terrain impliqués dès la phase discovery (sondage 12 IDEL) |

### 7.2 Risques non-techniques

- **Risque réglementaire** : une évolution des obligations HDS ou RGPD en cours de projet est peu probable sur 6 mois mais surveillée via veille mensuelle.
- **Risque de soutenance** : la date de soutenance est fixe et non négociable. Le scope MVP est dimensionné pour être livrable avec une marge de 2 semaines de tampon (phase 7). Les features post-MVP ne sont pas bloquantes pour la démonstration.
- **Risque académique** : les livrables documentaires (PRD, architecture, epics/stories, rapport) sont produits en parallèle du développement, pas après. Chaque artefact de planning est versionné dans le dépôt Git.

### 7.3 Plan de contingence

En cas de dérive de planning supérieure à 1,5 sprint, le plan de contingence prévoit :

1. Désactivation des features *Could have* (import CSV, export RGPD automatisé)
2. Simplification de l'algorithme TSP (version greedy uniquement, sans optimisation locale)
3. Report du back office web à une démonstration en read-only (données mockées)
4. Conservation absolue du mode offline, de l'authentification MFA et du planning de base — ces trois piliers sont non-négociables pour la démonstration terrain

## Conclusion

Si j'avais eu la responsabilité complète de l'organisation de ce projet, j'aurais structuré KURA autour de trois axes forts : une **méthode agile disciplinée mais allégée** (Scrum bi-hebdomadaire) adaptée à la réalité d'une équipe de 2, une **architecture de risque anticipée dès le sprint 0** (offline, HDS, sync, TSP), et une **implication régulière des utilisateurs terrain** aux moments à plus forte valeur ajoutée pour ne pas construire à l'aveugle.

La force du projet réside dans la combinaison d'une connaissance intime du métier (données terrain, proximité familiale avec le domaine IDEL) et d'une stack technique moderne répondant précisément aux contraintes identifiées. L'organisation décrite ici vise à protéger cette valeur en la livrant de façon fiable, dans les délais et les exigences réglementaires du secteur de la santé.
